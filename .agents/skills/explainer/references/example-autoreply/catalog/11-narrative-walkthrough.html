You have a live Ticket arriving in Zendesk and a question that has no safe answer today: what would the responder actually do to this? Running it for real is the only way to find out — and that means writes. It claims a lease in Dynamo, transitions the ticket, maybe files a draft, maybe sends the auto-reply, maybe stashes an audit blob in S3. You cannot preview the decision without committing to its consequences.

The way out was hiding in plain sight. never touches Dynamo, Zendesk, the draft, the auto-reply or S3 directly — every side effect already goes through a collaborator handed in from outside. If the writes are injected, then a preview isn't a new behaviour we bolt on. It's the same behaviour with different collaborators. There is no second code path to write, and therefore no second code path to drift out of sync.

That insight becomes one function. takes the real dependency bundle and returns a copy with every write collaborator swapped for a no-op double — while keeping the reasoning collaborators (classify, the draft brief, the reply template) and the real store's reads untouched. The reasoning stays honest; only the consequences are neutralised. Crucially, is never told any of this. It receives dependencies, as always, and runs.

The doubles are deliberately dull. passes reads straight through to the real store, so the run sees the real row, the real channel, the real IAM identity — but its claim and transition calls simply return truthy so the control-flow gates (if not store.claim(...)) pass and the one path runs end to end, persisting nothing. The for the rest — _NoopReply, _NoopDraft, _NoopStatus — log what they would have done and hand back a plausible-shaped value, so downstream rendering still works but no Zendesk write, no reply sent, and no draft filed ever happen.

So how does the caller ever see what happened, if nothing was written? That's the other half of the change. now packs everything it computed into a on the way out. A live run can still learn its outcome by observing the side effects it caused; a preview run reads the RunReport instead. The forks once at the very top on a direct invoke, renders that RunReport into a , and returns it. Two ways to read the same run — but still one code path producing it.

The subtle payoff is channel. A preview event you construct by hand could easily carry the wrong channel in its payload — but it doesn't matter, because never reads channel from the event. It reads it from the real ticket row, through 's pass-through get. The preview gets channel right for the same reason it gets everything else right: the reads are real. Only the writing is pretend.
